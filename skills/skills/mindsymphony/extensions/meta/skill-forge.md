# 技能熔炉 (Skill Forge)

> MindSymphony灵魂铸造 × skill-creator-meta痛点驱动 的融合版技能创建系统。

---


## 执行前四问

| 问题 | 本技能的检查点 |
|------|---------------|
| **目的** | 这个任务的最终交付物是什么？ |
| **调性** | 执行标准：快速完成/精细打磨？ |
| **约束** | 技术限制？格式要求？兼容性需求？ |
| **差异化** | 如何在「正确」基础上做到「优秀」？ |

**关键原则**：好的执行不是机械完成，而是在约束内追求最优解。

## Phase 0: 存在性审计

**目的**：决定是否需要创建这个技能

### 三个必答问题

```
1. 这个任务真的需要一个Skill吗？
   ├─► 是否有现成工具可以直接完成？
   ├─► 是否只是一次性任务？
   └─► 没有Skill时，Claude能做到多少？

2. 原始流程中哪些步骤是绕弯路？
   ├─► 每个步骤都是必要的吗？
   ├─► 探索过程中的试错应该删除还是保留？
   └─► 有没有可以合并或跳过的步骤？

3. 这个Skill的核心价值是什么？
   ├─► 如果只能保留一个文件，保留哪个？
   ├─► 最隐蔽的陷阱是什么？
   └─► 没有这个Skill，用户最可能在哪里卡住？
```

### 审计决策矩阵

| 检查项 | 是 | 否 |
|--------|----|----|
| 任务会重复执行多次？ | 继续 | 考虑放弃 |
| 存在隐蔽的技术陷阱？ | 必须包含 | 可简化 |
| 需要特定的决策逻辑？ | 必须包含 | 可简化 |
| 有现成工具可替代？ | 考虑放弃 | 继续 |
| Claude已经能做好80%？ | 只写20% | 正常构建 |

**输出**：一句话核心价值声明

```markdown
# 示例
✅ 好的核心价值："避免MiniMax TTS的HEX编码陷阱"
❌ 差的核心价值："提供完整的5阶段流程"
```

## Phase 1: 灵魂设计

**目的**：定义技能的哲学内核

### 必填项

| 要素 | 问题 | 示例 |
|------|------|------|
| **角色与使命** | 这个技能为何存在？ | "帮助用户避免统计方法误用" |
| **认知结构** | 它如何思考？ | 结构化分析型 / 发散创意型 / 严谨执行型 |
| **道法术器定位** | 它在系统中的层级？ | 道（价值）/ 法（战略）/ 术（方法）/ 器（执行）|

### 按类型决定人格设定

| 技能类型 | 判定标准 | 人格设定 |
|----------|----------|----------|
| **creative** | 需要创意、美学、情感 | ✅ 内在冲突 + 美学追求 + 独特视角 |
| **analytical** | 需要分析、判断、视角 | ⚠️ 独特视角 |
| **execution** | 需要流程、步骤、执行 | ❌ 纯方法论 |

### 可选项（仅creative类型）

| 要素 | 问题 |
|------|------|
| **内在冲突** | 它在哪两种倾向之间挣扎？ |
| **独特视角** | 它如何独特地看待世界？ |
| **美学追求** | 它追求什么样的美？ |

## Phase 2: 痛点聚焦

**目的**：识别真正让用户卡住的点

### 痛点分类

| 类型 | 特征 | 处理方式 |
|------|------|----------|
| **隐蔽陷阱** | 不报错但结果错误 | **必须详细记录** |
| **决策分歧** | 多个选择不知选哪个 | 提供决策树 |
| **配置繁琐** | 步骤多但不难 | 提供脚本自动化 |
| **概念困惑** | 不理解为什么 | 简短解释即可 |

### 核心原则

> **如果Claude不看这个Skill也能轻松完成，就不要写进去。**

### 痛点提取模板

```markdown
## 痛点清单

### 隐蔽陷阱 🔴
1. [陷阱描述]
   - 表现：[用户会看到什么]
   - 原因：[为什么会发生]
   - 解决：[正确做法]

### 决策分歧 🟡
1. [分歧点]

### 领域反模式（绝对禁止）

**切忌形式主义**：
- 模板化执行而不考虑场景
- 为了"看起来完整"而添加冗余
- 只管格式不管实质

**强制实用价值**：每个输出都应该能直接用，而非「还需要改改」


   - 选项A：[适用场景]
   - 选项B：[适用场景]
   - 推荐：[默认选择]

### 可省略 ⚪
- [Claude已经知道的内容]
- [不影响结果的细节]
```

## Phase 3: 最小化实现

**目的**：用最少的文档覆盖所有痛点

### 文件结构决策

```
你真正需要的文件
├── SKILL.md              # 必须：入口 + 核心内容
├── references/           # 按需：参考资料
│   └── [痛点相关].md     
└── scripts/              # 按需：自动化脚本
    └── [工具脚本].py

你不需要的文件
├── README.md             # 冗余：SKILL.md已包含
├── examples/             # 冗余：放在SKILL.md里
└── docs/                 # 冗余：references/已覆盖
```

### SKILL.md 模板

```markdown
---
name: [技能名]
description: "[一句话核心价值]。Use when (1)...(2)...(3)..."
---

# [技能名]

> [角色与使命]

## [仅creative类型] 创作人格
### 内在冲突
### 独特视角
### 美学追求

## 核心能力
[3-5个关键能力]

## 工作流程
[只写必要步骤，省略Claude已知内容]

## 痛点速查
### 🔴 隐蔽陷阱
### 🟡 决策分歧

## 标准输出模板
```

**目的**：将新技能整合到MindSymphony体系

### MECE归类

| 用户意图 | 归属模块 |
|----------|----------|
| "帮我规划..." | strategy/ |
| "帮我研究..." | research/ |
| "帮我设计..." | creative/ |
| "帮我写..." | writing/ |
| "帮我分析风险/多角度看..." | thinking/ |
| "帮我写代码..." | engineering/ |
| "帮我做自动化..." | domains/ |
| "帮我创建技能..." | meta/ |

### 触发词配置

```yaml
# config.yml
skill_router:
  trigger_patterns:
    - pattern: "[触发词1]|[触发词2]|[触发词3]"
      skill: "[模块]/[技能名]"
      priority: medium
      requires: [模块名]
```

### 协作设计

```yaml
# 这个技能会被谁调用？
upstream:
  - [上游技能]

# 这个技能会调用谁？
downstream:
  - [下游技能]
```

## Phase 5: 验证打包

### 验证检查清单

```bash
# 1. 结构验证
python3 scripts/validate_skill.py [skill_path]

# 2. 必须通过
✅ SKILL.md 存在
✅ frontmatter 格式正确
✅ name 和 description 存在
✅ description 包含 "Use when"

# 3. 建议通过
⚠️ 引用的文件都存在
⚠️ 没有冗余文件
```

### 打包输出

```bash
python3 scripts/package_skill.py [skill_path] [output_dir]

# 输出
[skill_name].skill  # tar.gz格式
[skill_name].zip    # zip格式
```
